

# üé® CSS Modules in Next.js ‚Äî Scoped, Simple, and Built-in

In modern web development, styling components in a scalable and modular way is crucial. Next.js supports **CSS Modules out of the box**, offering a way to write traditional CSS while keeping it **scoped to the component** ‚Äî avoiding global conflicts and promoting reusable styles.

This guide walks you through **how CSS Modules work in Next.js**, how to use them effectively, and why they‚Äôre perfect for component-based development.

---

## üß† What are CSS Modules?

**CSS Modules** are regular `.css` files where **each class or animation is scoped locally by default** ‚Äî meaning the styles only apply to the component that imports them.

```css
/* styles/Button.module.css */
.button {
  background-color: royalblue;
  color: white;
}
```

```js
// components/Button.js
import styles from './Button.module.css';

export default function Button() {
  return <button className={styles.button}>Click Me</button>;
}
```

> ‚úÖ Result: No className collisions. The styles are automatically scoped by Next.js.

---

## üèó File Naming Convention

To enable CSS Modules in Next.js, your CSS file must follow this naming pattern:

```
[ComponentName].module.css
```

Examples:
- `Button.module.css`
- `Navbar.module.css`

Next.js will throw an error if you try to import a non-module CSS file as a module.

---

## üîß How CSS Modules Work

When you import a CSS Module file, Next.js compiles and transforms the class names into **unique identifiers**, like:

```css
/* Button.module.css */
.button {
  background: royalblue;
}
```

gets compiled into:

```html
<button class="Button_button__xYz8k">Click Me</button>
```

So it won‚Äôt conflict with other `.button` styles elsewhere in your app.

## üéØ Using Multiple Classes

```css
/* styles/Card.module.css */
.card {
  padding: 1rem;
}
.highlight {
  border: 2px solid red;
}
```

```js
import styles from './Card.module.css';

<div className={`${styles.card} ${styles.highlight}`}>Hello</div>
```

Or with [clsx](https://github.com/lukeed/clsx) or `classnames` package:

```bash
npm install clsx
```

```js
import clsx from 'clsx';
<div className={clsx(styles.card, styles.highlight)} />
```

## üé® Global Styles vs CSS Modules

| Feature              | Global CSS (`styles.css`) | CSS Modules (`.module.css`)       |
|----------------------|---------------------------|------------------------------------|
| Scope                | Global                    | Component-level (scoped)           |
| Naming Conflicts     | Possible                  | Avoided (auto-generated names)     |
| Reusability          | Low                       | High                               |
| Next.js Support      | Must be imported in `_app.js` | Can be imported in any component |

‚úÖ Use **CSS Modules** for component styles  
‚úÖ Use **global styles** for resets, typography, and base layout

---

## üì¶ How to Import Global Styles

```js
// pages/_app.js
import '../styles/globals.css';
```

But **don‚Äôt import global CSS inside components** ‚Äî Next.js doesn‚Äôt allow that.

## üßë‚Äçüíª Real-world Example

### File: `components/ProfileCard.js`

```js
import styles from './ProfileCard.module.css';

export default function ProfileCard({ name, bio }) {
  return (
    <div className={styles.card}>
      <h2>{name}</h2>
      <p>{bio}</p>
    </div>
  );
}
```

### File: `components/ProfileCard.module.css`

```css
.card {
  background: #fff;
  border-radius: 10px;
  padding: 20px;
  box-shadow: 0 2px 5px rgba(0,0,0,0.1);
}
```

Boom! Clean, isolated, reusable component styling!

## üß† Best Practices

- Use CSS Modules for **individual UI components**
- Keep class names **semantic and short** (`btn`, `title`, `card`)
- Use **utility-first frameworks like Tailwind** for global utility classes (optional)
- Avoid nesting too deeply in `.module.css`
