

#  **Checked vs Unchecked Exceptions**

---

## ЁЯФН Overview

In Java, exceptions are broadly classified into:

* **Checked Exceptions** тАУ enforced at **compile-time**
* **Unchecked Exceptions** тАУ occur at **runtime**, i.e., `RuntimeException` and its subclasses

This distinction is critical to **API design**, **error resilience**, and **developer experience** тАФ and often features in **interviews**, where design decisions and trade-offs are deeply analyzed.

## ЁЯФ╖ 1. **Checked Exceptions** (Compile-Time Enforced)

### ЁЯУМ Definition:

Exceptions that must be either **caught** or **declared** in the method signature using `throws`.

### ЁЯФз Use Case:

When **client code must take an explicit recovery action**, e.g., file not found, network error, DB failure.

### ЁЯТб Common Checked Exceptions:

* `IOException`
* `SQLException`
* `ClassNotFoundException`
* `ParseException`

### ЁЯФм Code Example:

```java

    Files.readAllLines(Path.of(path));
}
```

OR

```java
try {
    readFile("config.txt");
} catch (IOException e) {
    // Handle error: show error message, fallback to defaults
}
```

## ЁЯФ╢ 2. **Unchecked Exceptions** (Runtime Exceptions)

### ЁЯУМ Definition:

Exceptions that are **not checked at compile time** and **do not require explicit handling**.

### ЁЯФз Use Case:

Represent **programming errors or logic violations** тАУ things that should never happen if the code is correct.

### ЁЯТб Common Unchecked Exceptions:

* `NullPointerException`
* `IllegalArgumentException`
* `ArrayIndexOutOfBoundsException`
* `ArithmeticException`
* `IllegalStateException`

### ЁЯФм Code Example:

```java

    if (b == 0) {
        throw new IllegalArgumentException("Divisor cannot be zero");
    }
    return a / b;
}
```

No need to declare `throws` or force caller to catch this.

## ЁЯза When to Use What?

| Scenario                                                   | Exception Type | Rationale                      |
| ---------------------------------------------------------- | -------------- | ------------------------------ |
| API expects caller to fix issue                            | Checked        | Enforce compile-time awareness |
| Caller should not be burdened                              | Unchecked      | Keep API cleaner               |
| Recoverable business logic failure (e.g., retry, fallback) | Checked        | Force user to handle           |
| Programmer error (null, illegal state)                     | Unchecked      | Encourage fail-fast            |

## тЪая╕П Drawbacks of Checked Exceptions

* тЭМ **Cluttered method signatures**

  * Especially in deeply layered systems (controller тЖТ service тЖТ DAO)
* тЭМ **Forces wrapping or propagation**

  * Often leads to boilerplate: `try { ... } catch (...) { throw new RuntimeException(...) }`
* тЭМ **Discouraged in functional or reactive paradigms**

  * Lambdas don't work well with checked exceptions

## тЬЕ Best Practices for Exception Usage (Enterprise-level)

| Do                                                                   | DonтАЩt                                                                                |
| -------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| тЬЕ Use checked exceptions for recoverable system errors               | тЭМ DonтАЩt overuse them for simple validation                                           |
| тЬЕ Convert low-level checked exceptions to domain-specific exceptions | тЭМ DonтАЩt expose implementation exceptions (like `SQLException`) directly to UI layers |

| тЬЕ Use custom exception hierarchy (e.g., `AppException`)              | тЭМ DonтАЩt swallow exceptions without logging or reporting                              |

## ЁЯТм Interview Insight

**Q:** *When would you use a checked exception vs unchecked?*

**A:**

> If a method exposes a scenario where the caller is expected to handle a recoverable situation тАФ like retrying a failed file read or handling business rule violations тАФ I use a **checked exception** to enforce it.
>
> But for internal logic bugs or improper usage тАФ like null arguments or invalid states тАФ I prefer **unchecked exceptions**, as they reflect coding errors that should be fixed, not caught.
