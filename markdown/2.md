In this article, I am going to cover the topic of software snapshot testing using **Jest**. You may not realise the significance of snapshot testing unless you have ever worked on a large-scale corporate project or any well-known crowded open source project.

**What is Snapshot testing?**

On a CI system, it is always recommended to always commit all snapshots and to keep them in version control. Snapshots represent the state of the source modules at a given moment. So when state modules are changed, Jest can tell what changed from the previous version. This helps code reviewers study your changes better. The snapshot testing is specifically used to check whether the output is correct. Apart from React and React Native, snapshot testing can be applied to any value that is serializable. Unlike visual regression testing, which takes screenshots and compares images pixel by pixel, snapshot testing compares serialised values kept in text files using a diff algorithm. Hence, snapshot testing is often called "gloden master technique" or "output comparison testing."


Snapshots are a tool used in unit testing. If you don't use snapshots, then you end up with large chunks of test data in your unit test code. For example, imagine you are testing a function that returns a hundred lines of text, like a blog post. One way would be to have an `expect(blogPost).toBe(thisHugeStringOfManyLines)`. We can use snapshots to easily manage large chunks of test data. Jest creates separate files for test data and provides an easy-to-use diffing mechanism to see what parts of the data have changed. In the case of react and enzyme, the testing data is JSX.

**Example**

Let me create and start a React app first. This tool chain already comes with pre-installed Jest.


    npx create-react-app my-app 
    cd my-app 
    npm start
    npm run test
You will notice a file `/src/__snapshots__/App.test.js.snap` has been created after running the `npm run test` .

  **Now what is happening here?**
  

 JEST renders the UI component, takes a snapshot, and then compares it to the previous snapshot file stored alongside the test. The test failed because the two snapshots didn't match; either the change was unexpected or the reference snapshot needs to be updated to a new version of the UI component.


Let's update the reference snapshot by pressing `w` to show more options, then pressing `u` to update. Now your snapshot test case will pass because the reference snapshot is updated as per the updated UI.
